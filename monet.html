01

다형성을 활용한 오브젝트 구조

기획 요구사항

Requirements
  • 레이에 닿은 모든 상호작용 가능한 오브젝트에 아웃라인 셰이더를 입힌다.
  • 오브젝트 크기, 퍼즐의 특성을 고려해 오브젝트마다 상호작용이 가능한 거리를 조절해야 한다.
  • 특정 오브젝트는 인벤토리에 넣거나 뺄 수 있다.
  • 특정 오브젝트는 잡았을 경우 혹은 놓았을 경우 특정 기능을 수행한다. (크기 착시 효과, 포지션 리셋, 소켓에 장착 등)

설계 및 구현

상속을 통한 코드의 재사용성 — 모든 오브젝트들이 수행해야 할 기능을 상위 오브젝트에 정의하고 이를 상속하여 다양한 오브젝트를 제작하도록 하였습니다.

Virtual 키워드를 통한 재정의 — 같은 행동(SelectEnter : 잡혔을 때 호출, SelectExit : 놓였을 때 호출)에서 다양한 기능을 수행할 수 있도록 가상 함수로 구현했습니다.

InteractObject 클래스에는 Hover(RayInteract에 트리거)된 경우 아웃라인 셰이더를 그리는 UpdateOutLine 메서드를 정의하였고, 오브젝트와 플레이어 간의 상호작용할 수 있는 최소 거리, 상호작용이 된 경우 플레이어로부터 오브젝트가 위치하는 거리를 정의하였습니다.

추가로, OnSelectEntering(), OnSelectExiting() 등 Virtual로 선언된 메서드의 다형성을 활용하여 다양한 기능을 수행하는 오브젝트를 제작하였습니다.

02

ResizingObject를 상속한 크기 착시 오브젝트

핵심 기획 (Superliminal 스타일 기믹)

해당 기믹은 잡은 오브젝트에 원근이라는 개념을 부여하지 않고, 화면에 그려지는 크기 그대로 놓였을 때도 유지하는 것입니다. 실제 구현한 오브젝트는 스테이지 1 체스 퍼즐입니다.

구현 메커니즘 (Resize 함수)

오브젝트를 잡기 시작한 순간 오브젝트와 카메라 사이의 거리를 최초 거리로 할당합니다.

이후 카메라 정면으로 레이(Ray)를 발사해 벽을 체크합니다.

오브젝트를 닿은 벽의 앞부분에 위치시키고 다시 한번 거리를 체크, 최초 거리와의 비율을 계산해 오브젝트의 크기를 변경합니다.

결과적으로 유저가 최초 거리보다 멀어질수록 오브젝트는 커지고, 최초 거리보다 가까워질수록 오브젝트는 작아집니다. 이를 통해 화면상으로 오브젝트의 크기는 변하지 않는 듯한 착시 효과가 발생합니다.

03

인벤토리 시스템과 ItemObject

3-1. MVC 패턴의 활용

인벤토리의 데이터와 UI를 MVC 패턴을 응용하여 구현하였습니다.

  • 아이템 데이터는 고유의 ID 값을 가지며, 이 ID는 아이템 관리자(ItemManager)의 Dictionary에서 Key 값으로, ItemData는 Value 값으로 할당하여 사용합니다.
  • 데이터 관리자(DataManager)에 의해 관리되는 인벤토리 슬롯은 실제 오브젝트를 소유하고 있는 것이 아닌 ItemData의 ID를 소유하며, 슬롯이 비어있는 경우 이는 -1 값으로 초기화됩니다.
  • 아이템 삽입 시 아이템 객체는 삭제되고 슬롯에 아이템의 ID가 할당됩니다.
  • 슬롯은 인벤토리를 켰을 때 활성화(OnEnable)되며, 이때 데이터 매니저를 통해 할당된 아이템의 ID 값을 확인하고 해당 아이템의 아이콘을 UI에 출력합니다.

ItemID 값이 변경됨에 따라 실행될 이미지 변경 기능을 set 프로퍼티를 사용해 구현하였습니다.

3-2. 오브젝트 풀링

인벤토리 시스템은 아이템을 넣고 꺼내는 행위가 잦아 객체의 생성과 삭제가 빈번하게 일어납니다. 메모리의 효율적인 관리와 최적화를 위해 오브젝트 풀링(Object Pooling)을 적용했습니다.

데이터 테이블을 통해 아이템의 데이터를 불러올 때, 아이템 데이터에서 참조하는 실제 아이템 객체들을 풀링하기 위한 풀(Pool)을 생성합니다.

아이템 풀의 기본 크기를 정해놓고 Queue 자료구조의 Enqueue와 Dequeue를 사용해 오브젝트를 활성화/비활성화하도록 구현하였습니다. 이를 통해 인벤토리 조작으로 인한 가비지 컬렉션(GC) 과부하를 줄이고 데이터를 효율적으로 관리했습니다.
